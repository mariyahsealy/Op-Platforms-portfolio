 Project Overview

This repository contains the completed software design document for The Gaming Room, a client seeking to expand their existing Android game application, Draw It or Lose It, into a web-based, multi-platform experience. The Gaming Room wanted a distributed application accessible across Windows, macOS, Linux desktop browsers, as well as iOS and Android mobile devices, while maintaining the core gameplay mechanics from their original Android version. The game, similar to the 1980s television show Win, Lose or Draw, involves teams competing to guess puzzles as stock drawings render progressively over timed rounds. The client's requirements included supporting multiple concurrent game sessions, ensuring unique game and team names, implementing secure user authentication with role-based access control, and guaranteeing that only one instance of the game exists in memory at any given time through proper design patterns.

 Reflection on Documentation Development

One of my strongest contributions to this documentation was the comprehensive evaluation table comparing operating platforms across server-side hosting, client-side compatibility, and development tool requirements. I provided specific cost analyses, licensing details, and technical trade-offs for each platform, giving The Gaming Room concrete data to inform their infrastructure decisions. For instance, I quantified that Linux server hosting costs approximately $30-50/month on AWS compared to $780/month for equivalent macOS instances, making the cost-benefit analysis immediately clear. Additionally, I thoroughly documented the application architecture using UML diagrams that clearly illustrated the Entity base class inheritance structure, the GameService singleton pattern, and the composition relationships between Game, Team, and Player classes. These diagrams, combined with detailed explanations of design pattern implementations, provided both technical depth for the development team and accessible clarity for non-technical stakeholders at The Gaming Room.


Working through the design document before writing code proved invaluable in several ways. First, the domain modeling exercise—creating the UML class diagram showing Entity as the base class with Game, Team, and Player inheriting from it—clarified the object-oriented structure before I wrote a single line of Java. This upfront design prevented refactoring later; I knew exactly which attributes belonged in the base class (id and name) and which were specific to subclasses (teams list in Game, players list in Team). Second, explicitly documenting the singleton pattern's purpose and implementation in the design constraints section forced me to think through *why* GameService needed to be a singleton (preventing duplicate game instances and ID conflicts) rather than just mechanically implementing the pattern. This deeper understanding led to more thoughtful code with comprehensive comments explaining the pattern's role. Finally, articulating the RESTful API design and authentication requirements in the document created a clear specification that guided the GameAuth implementation. When I later wrote the GameAuthenticator and GameAuthorizer classes, I referred back to the design document's security section to ensure I properly implemented role-based access controls restricting certain endpoints to ADMIN or USER roles as specified.


If I could revise one aspect of this work, I would enhance the Recommendations section by including a more detailed cloud migration roadmap with specific milestones and timelines. While I recommended AWS with containerized deployment using ECS/EKS, I provided limited guidance on the transition process from The Gaming Room's current Android-only infrastructure to a cloud-based, multi-platform system. An improved version would include a phased approach: Phase 1 would be develop and deploy minimal viable product with core gameplay on AWS with 2-3 game sessions supported; Phase 2 would be implement auto-scaling, CDN integration, and expand to 50+ concurrent games; Phase 3 would be add advanced features like real-time leaderboards, player profiles, and matchmaking. I would also provide more specific cost projections at each phase, including monthly AWS bills for development, staging, and production environments, as well as estimated developer hours required. This would give The Gaming Room a clearer understanding of both financial investment and timeline, making the path from current state to fully deployed web application more tangible and actionable.


I interpreted the user's needs by carefully analyzing both explicit requirements and implicit requirements. The most critical user need was cross-platform accessibility—players expect to start a game on their iPhone during their commute, continue on their laptop at home, and maintain consistent experience across devices. I implemented this by designing a web-based architecture with responsive HTML/CSS front-ends that adapt to screen size, backed by a RESTful API that any platform can consume using standard HTTP protocols. For the unique name requirement, I implemented the iterator pattern in GameService.addGame(), Team.addTeam(), and Player.addPlayer() methods that search existing collections before creating new instances, directly preventing the frustrating user experience of duplicate names causing confusion or data corruption.

Considering user needs during design is crucial because the application's technical elegance is irrelevant if it doesn't solve real user problems. For example, I could have implemented the most sophisticated caching algorithm, but if images still took 10 seconds to load on mobile devices over cellular networks, users would abandon the game in frustration. By prioritizing user needs—30-second image rendering deadline, mobile-friendly interfaces, secure authentication that doesn't require remembering complex passwords—I ensured the technical architecture served the actual user experience. Additionally, understanding that The Gaming Room's users are casual gamers, not technical experts, influenced security design: implementing Basic Authentication with clear browser dialogs rather than complex OAuth flows that might confuse non-technical players. Every technical decision, from choosing JSON as the data format to recommending AWS regions near major population centers, traced back to improving user experience.

 Software Design Approach and Future Strategies

My approach to designing this software began with requirements analysis carefully reading the client's specifications to extract both functional requirements and non-functional requirements like performance, security, scalability. I then employed **domain modeling** to identify core entities like Game, Team, Player, User and their relationships, which informed the object-oriented class structure. Next, I evaluated design patterns appropriate for the requirements: the singleton pattern for GameService to enforce single-instance constraint, the iterator pattern for duplicate name prevention, and the factory pattern for controlled object creation. I also conducted platform evaluation, systematically comparing Linux, macOS, Windows, and mobile platforms across server capabilities, client compatibility, development tools, costs, and trade-offs to make evidence-based infrastructure recommendations.

For future projects, I would employ several additional techniques. First, I would create user journey maps that visualize the end-to-end player experience from account creation through game completion, identifying pain points and opportunities for optimization at each step. Second, I would develop architectural decision records that formally document key design choices with context, considered alternatives, and rationale, creating an auditable trail of design thinking. Third, I would incorporate threat modeling earlier in the design process, systematically identifying security vulnerabilities and designing defenses proactively rather than reactively. Fourth, I would use prototyping and iterative validation by building minimal proof of concept implementations of risky architectural components to validate technical feasibility before committing to full design. Finally, I would engage in stakeholder workshops with both The Gaming Room's business team and representative end users, using wireframes and mockups to validate that the proposed design meets actual needs before extensive development begins. These strategies collectively reduce risk, improve design quality, and ensure alignment between technical implementation and business objectives.
